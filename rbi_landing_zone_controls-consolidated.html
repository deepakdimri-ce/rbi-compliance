◆ IDENTITY & ACCESS
Identity
Cloud Identity / Identity Federation
 Centralised identity directory or federation with corporate IdP (AD/LDAP) via SAML 2.0 / OIDC. Enables audited, need-to-know access governance across all GCP resources.
→ RBI Outsourcing Guidelines 2006 §5.6.2 (need-to-know access) · §5.6.1 (confidentiality & security)

IAM — Least Privilege & Separation of Duties
 Granular IAM roles; no standing owner-level access in production; separation of duties between dev, ops, and security. Cited verbatim in Google's RBI Data Localisation whitepaper as a required technical control.
→ RBI Outsourcing Guidelines 2006 §5.6.2 (need-to-know) · §5.6.3 (isolation per bank) · cited by name in Google's Data Localisation whitepaper
 Cloud IAM RBI Required

 Enforce MFA for all users accessing GCP. Aligned to RBI's cybersecurity framework for banks which broadly mandates strong authentication controls. Not cited as a GCP-specific control in Google's RBI mapping but directly supports the spirit of RBI's authentication requirements.

 Just-in-time privileged access elevation with time-bound, justification-required grants. Eliminates standing privileged access — significantly reduces insider threat and compromised credential blast radius. Google MVSP Advanced tier recommendation.

Org Policy — India Region Restriction
 Enforce constraints/gcp.resourceLocations restricting ALL GCP resources to asia-south1 (Mumbai) and asia-south2 (Delhi NCR) only. The single most critical technical control for RBI data localisation compliance.
→ RBI/2017-18/153 — Storage of Payment System Data (DPSS.CO.OD No.2785/06.08.005/2017-18) · cited explicitly in Google's Data Localisation whitepaper as the primary enforcement mechanism
 Org Policy Service RBI Required
Cited verbatim as the primary technical control in Google's RBI Data Localisation whitepaper.

Resource Hierarchy (Org → Folders → Projects)
 Structure GCP as Organisation → Folders (Production / Non-Production / Shared Services / Security) → Projects. Foundational prerequisite — without this, IAM, org policies, and audit scope cannot be applied consistently across the bank.
→ RBI Outsourcing Guidelines 2006 §5.6.3 (logical isolation of bank's data) — structural foundation for all other technical controls
 Resource ManagerFolders RBI Required
Structural prerequisite for isolating the bank's data per §5.6.3. Must be defined before any workload deployment.

→ RBI Outsourcing Guidelines 2006 §5.6.2 (need-to-know oversight of Google access) · §5.9.1 (monitoring) — Access Transparency named by product in Google's RBI outsourcing mapping
 Access TransparencyAccess Approval RBI Required
Named by product in Google's RBI mapping §5.6.2 and §5.9.1. Enables oversight of Google's own access to bank data.

◆ NETWORK SECURITY
Network
VPC Service Controls
 Create service perimeters around GCS, BigQuery, Cloud SQL, and Spanner to prevent data egress outside the India boundary. Named by product in Google's RBI Data Localisation whitepaper alongside Org Policy as the two required technical controls.
→ RBI/2017-18/153 — Storage of Payment System Data · cited by product name in Google's Data Localisation whitepaper as the data egress enforcement layer
 VPC Service Controls RBI Required
Named explicitly in Google's RBI whitepaper as one of two required technical controls for data localisation alongside Org Policy.
 Docs
Network
Shared VPC Architecture
 Centralised network management via a Shared VPC host project. All service projects consume network resources from a single bank-governed hub. Google's recommended architectural pattern to satisfy the §5.6.3 isolation requirement — not prescribed by name in RBI documents.
→ RBI Outsourcing Guidelines 2006 §5.6.3 (isolation) — Shared VPC is Google's recommended implementation pattern
 Shared VPC Google Recommends for RBI
Google's recommended architecture for §5.6.3 isolation. Not named in RBI documents but directly enables compliance.

Network
Cloud Interconnect (hybrid deployments)
 Dedicated private connectivity between the bank's on-premises DC and GCP India regions. Required when Core Banking System or other on-prem workloads communicate with GCP. Google's RBI BC/DR guidance requires robust continuity frameworks — Interconnect is the enabler for hybrid architectures.
→ RBI Outsourcing Guidelines 2006 §5.8 (BC/DR) · RBI Master Direction IT Outsourcing 2023 Chapter VI
 Cloud InterconnectPartner Interconnect Google Recommends for RBI
Conditional — only required for hybrid banking deployments with on-prem systems connecting to GCP.

Hierarchical Firewall Policies
 Define firewall policies at Organisation and Folder level that take precedence over project-level rules. Enforce deny-all default with explicit allow lists. Prevents developers from inadvertently opening insecure firewall rules in production banking workloads.

Google MVSP Intermediate. Critical for preventing uncontrolled firewall rule sprawl across banking project teams.

Cloud Armor — WAF & DDoS Protection
 Deploy Cloud Armor in front of all internet-facing banking applications (internet banking portal, mobile APIs). Configure OWASP Top 10 WAF rules, rate limiting, and adaptive DDoS protection. Critical for protecting customer-facing digital banking channels.
→ Google Cloud MVSP — Network Security · RBI Cybersecurity Framework Annex 1 (application layer protection)

Google MVSP + RBI Cybersecurity Framework Annex 1. Essential for internet-facing banking applications.

◆ DATA PROTECTION & ENCRYPTION
Data
Default Encryption at Rest & in Transit
 All GCP data encrypted at rest (AES-256) and in transit by default. Row 26 of Google's RBI outsourcing mapping lists this as the confidentiality baseline meeting §5.6.1. No configuration needed — the bank must confirm it is active and document it as RBI audit evidence.
→ RBI Outsourcing Guidelines 2006 §5.6.1 (confidentiality & security) · Row 26 of Google's RBI outsourcing mapping — "security by default" baseline
 Google-managed encryption RBI Required
Cited in Row 26 of Google's RBI mapping as the default confidentiality baseline. On by default — confirm and document for audit.

Data
Cloud KMS — Customer-Managed Keys (CMEK)
 Bank-controlled encryption keys for all financial and payment data. Google's RBI Data Localisation whitepaper cites Cloud KMS by product name for "sensitive data such as customer financial information, credit card numbers and personal identifiers" — enabling the bank to revoke access independently of Google.
→ RBI Outsourcing Guidelines 2006 §5.6.1 (confidentiality) · Cloud KMS cited by product name in Google's Data Localisation whitepaper
 Cloud KMSCMEK RBI Required
Cloud KMS cited by product name in Google's RBI whitepaper specifically for banking/payment sensitive data protection.

Cloud HSM for Critical Key Material
 FIPS 140-2 Level 3 certified Hardware Security Modules for payment signing keys, CBS encryption, and certificate issuance. HSM keys are physically located within India (asia-south1). Satisfies advanced key custody requirements for payment processors.
→ Google Cloud MVSP — Data Protection (Advanced tier) · payment system HSM requirements
 Cloud HSMCloud KMS Best Practice
Google MVSP Advanced. Not RBI-required but recommended for payment HSM-specific workloads and high-value signing keys.

Sensitive Data Protection (DLP)
 Discover, classify, and de-identify PII and financial data across GCS, BigQuery, and Cloud SQL. Apply de-identification for non-production environments. Enforce data classification labels. Essential for DPDPA 2023 compliance and overall data governance for the bank.
→ Google Cloud MVSP — Data Protection · RBI Cybersecurity Framework (data classification) · DPDPA 2023
 Sensitive Data ProtectionCloud DLP Best Practice
Google MVSP + DPDPA 2023. Important for protecting customer PII — handle as a parallel workstream alongside RBI MVP.

 Store all application secrets, API keys, database credentials, and certificates in Secret Manager with automatic rotation. Prevents hardcoded credentials in code or config files — a common audit finding when banks migrate to cloud.

◆ LOGGING & MONITORING
Logging
Cloud Audit Logs + Cloud Operations
 Enable Admin Activity and Data Access audit logs across all projects. Google's RBI mapping §5.9.1 cites Google Cloud Operations by product name as the required monitoring tool for RBI outsourcing oversight. Audit logs are the bank's primary evidence for RBI supervisory reviews.
→ RBI Outsourcing Guidelines 2006 §5.9.1 (monitoring & control) — Google Cloud Operations cited by product name in Google's RBI mapping
 Cloud Audit LogsCloud Operations RBI Required
Cloud Operations cited by product name in §5.9.1 of Google's RBI mapping. Enable at org level across all projects.
 Docs
Logging
Log Export & Long-term Retention (India)
 Export audit logs to a GCS bucket in India region (asia-south1 / asia-south2). Retain for minimum 5 years per RBI norms. Data localisation rules apply to audit logs — they cannot be stored outside India. Google's RBI mapping §5.5.1 requires data preservation per the bank's legal/regulatory obligation.
→ RBI Outsourcing Guidelines 2006 §5.5.1 (data preservation obligation) · RBI/2017-18/153 data localisation applies to audit logs
 Cloud StorageLog Sinks RBI Required
Data preservation cited in §5.5.1. Logs must remain in India per localisation directive and be retained per RBI audit norms (5 years min).

 Organisation-wide threat detection, misconfiguration alerts, vulnerability findings, and compliance posture reporting. SCC Premium provides compliance reports mapped to CIS, PCI-DSS, ISO 27001, and NIST 800-53 frameworks which can be used as supporting evidence for RBI audits.
→ Google Cloud MVSP — Monitoring, Logging & Alerting (Intermediate tier) · supports RBI §5.9.1 ongoing monitoring
 Security Command Center Best Practice

GCS Bucket Lock / WORM for Audit Logs
 Apply Bucket Lock (Write Once Read Many) and object retention policies to GCS buckets holding audit logs. Prevents tampering or deletion of audit evidence. Supports the immutability of audit trails expected by RBI during supervisory inspections.
→ Google Cloud MVSP — Monitoring, Logging & Alerting · supports RBI audit trail integrity requirement

Google best practice for audit log integrity. Ensures logs cannot be deleted or modified before the retention period expires.

 Centralised security log analysis, threat detection rules (YARA-L), and automated incident response playbooks. Chronicle retains all telemetry in India region. Enables the 24×7 Security Operations Centre capability that RBI's Cyber Resilience framework requires banks to maintain.

OS Patch Management & Shielded VMs
 Automated patching of all Compute Engine VMs via OS Config. Enforce Shielded VMs with Secure Boot and vTPM for integrity monitoring. RBI's Cybersecurity Framework requires timely patching as a baseline cyber hygiene control for all banks.
→ Google Cloud MVSP — Infrastructure (Basic tier) · RBI Cybersecurity Framework Annex 1 §4.3 (patch management)
 VM ManagerOS ConfigShielded VMs Best Practice
Google MVSP Basic + RBI Cybersecurity Framework §4.3. Timely patching is a baseline RBI cyber hygiene requirement.

Infrastructure
Binary Authorization & Container Security
 Enable Binary Authorization on GKE to only allow signed, verified container images from trusted registries to run in production. Integrate with Artifact Registry vulnerability scanning. Prevents deployment of compromised or unapproved software in banking environments.
→ Google Cloud MVSP — Infrastructure (Advanced tier) · supply chain security for containerised banking workloads

Google MVSP Advanced. Important for containerised banking workloads — prevents rogue images running in production.

IaC & CI/CD Pipeline Security
 All infrastructure deployed via Terraform through a secured CI/CD pipeline. No manual console changes in production. Enforce policy-as-code checks (e.g. Terraform Validator) before deployment. Supports RBI's change management and audit trail requirements.
